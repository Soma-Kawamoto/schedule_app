# 🚀 Git Rebase Master Cheatsheet
**「歴史を綺麗に、そして一直線に。」**
rebase は、ブランチの分岐元（ベース）を別の場所に付け替えることで、コミット履歴を一直線に整理する強力なコマンドです。
## 📖 1. Rebaseの基本概念 (Mergeとの違い)
機能追加（feature ブランチ）が終わって、最新の main ブランチを取り込みたい時、2つのアプローチがあります。
### 🌿 Merge（マージ）
- 特徴: 2つのブランチの歴史をそのまま残し、合流点（マージコミット）を作る。
- メリット: 実際に何が起きたか、過去の歴史が正確に残る。
- デメリット: ブランチが複雑に交差し、履歴（ネットワーク図）が「路線図」のようにごちゃごちゃしやすい。
### 🪡 Rebase（リベース）
- 特徴: feature ブランチの根元を、最新の main の先端に**付け替える**。
- メリット: 履歴が「一直線（1本の線）」になり、誰が見ても変更の流れが追いやすくなる。
- デメリット: 過去のコミットを新しく作り直す（歴史改変）ため、使い方を間違えると混乱を招く。

## ✨ 2. Interactive Rebase (-i) : 
歴史のタイムマシンrebase に -i (interactive) オプションをつけると、過去のコミットを自由自在に操る「神のモード」に入れます。
```Bash
# 直近3つのコミットを整理する
git rebase -i HEAD~3
```

### 📝 よく使うコマンド（Vim等のエディタで指定）
コマンド省略意味・アクションpickpそのコミットをそのまま使う（デフォルト）rewordrコミットは残すが、メッセージを書き換えるediteコミットの中身自体を修正・分割するsquashsこのコミットを、1つ前のコミットに融合（合体）させるdropdこのコミットを歴史から完全に消し去る

## 🛠️ 3. 実践：綺麗なSquashの流れ
細かい「WIP（作業中）」コミットを1つにまとめてから、リモートにプッシュする王道のフローです。
```Bash
# 1. 作業ブランチで細かいコミットを重ねる
git add .
git commit -m "ボタン追加"
git commit -m "typo修正"
git commit -m "スタイル微調整"

# 2. 直近3つのコミットを1つにまとめる（Squash）
git rebase -i HEAD~3
# => エディタが開き、2つ目と3つ目を 'squash' にして保存

# 3. リモートへ強制プッシュ（自分のブランチのみ！）
git push --force origin feature/my-branch
```

## ⚠️ 4. 【絶対厳守】Rebaseの黄金ルール
**🔥 「他人のいる歴史は、絶対に改変してはならない」 🔥**
自分が1人で作業しているブランチ（リモートにあっても、自分しか触っていないならOK）での rebase や squash は自由に行ってください。しかし、main ブランチや、チームメンバーと共有しているブランチで rebase を行い、push --force をすることは絶対にNGです。他人の作業履歴を消し去る大事故に繋がります。
